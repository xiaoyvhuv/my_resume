让上下文更“轻”：面向前端产品的 LLM 记忆分层策略

并不是把所有历史对话都塞给模型，体验就会更好。对产品来说，更重要的是知道哪些信息应该保留、哪些应该压缩、哪些应该被遗忘。

Memory Context UX

短期记忆不等于完整聊天记录

许多 LLM 应用默认把整段聊天记录原样提交给模型，但这会很快带来上下文冗余。对前端产品来说，真正需要保留的是当前任务中仍然“活跃”的信息，而不是所有历史痕迹。

所以我更倾向于把短期记忆理解为“当前会话工作区”。它可以包含最近几轮对话、当前任务目标以及用户最新提供的数据，而不是毫无筛选的长上下文。

长期记忆适合保存偏好，而不是临时任务

我在设计记忆层时，会把用户长期偏好单独抽出来，例如语言风格、输出格式偏好、常见主题兴趣等。这类信息更新频率低，但对体验提升非常稳定。

相反，像“今天帮我整理面试问题”这样的任务性信息，不应该进入长期层，否则模型很容易在未来不合时宜地召回旧目标。

摘要层是连接两者的桥梁

当短期会话变长时，我通常不会简单截断，而是先生成结构化摘要。摘要不是为了节省字符那么简单，它更像是把会话压缩成模型真正可用的“任务状态”。

  • 当前用户目标是什么。
  • 已经确认的约束有哪些。
  • 还未解决的问题是什么。

这样一来，系统就能把对话从“历史文本”转成“任务快照”，这对前端交互设计也更友好，因为它天然适合可视化展示。
