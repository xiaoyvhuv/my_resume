从 Prompt 到 Workflow：我如何搭建一个可复用的 LLM 前端实验台

对我来说，真正好用的 LLM 工具不是“能问答”就够了，而是应该具备可复现、可比较、可沉淀的实验结构。这个实验台就是基于这个目标逐步长出来的。

Prompt Workflow Experiment

为什么我不满足于一个聊天框

最早我也只是把模型接进一个最普通的对话框里，但很快就发现这种方式几乎无法支撑严肃的探索。一旦我开始尝试不同 prompt、不同 system instruction、不同上下文拼接策略，就很难回头比较哪一种方案真正有效。

于是我把目标改成了“实验台”而不是“聊天框”。前者强调的是过程记录，后者强调的是一次性结果。这个转变对界面结构影响很大。

我如何拆分这套前端结构

整个界面被我拆成四个核心区域：

  • 输入区：负责 prompt、system instruction 与变量模板。
  • 上下文区：显示检索片段、历史摘要与用户偏好。
  • 结果区：不仅展示答案，也展示中间状态与 token 信息。
  • 对比区：允许我把不同版本的 prompt 放在一起做横向比较。

这种布局带来的好处是，所有影响模型输出的因素都被显式化了。对前端来说，这意味着状态管理更复杂，但认知负担反而更低。

Prompt 不该只是字符串

当我把 prompt 当成配置对象，而不是单纯文本后，整个系统的可维护性有了明显提升。一个 prompt 模板里会包含角色设定、输入槽位、可选上下文、输出格式约束与版本号。

这样做的价值在于：当模型效果变化时，我可以明确知道到底是模板变了、上下文变了，还是输出解析方式变了。对于长期积累来说，这种边界感非常重要。

一个更接近产品的工作流视角

当系统开始支持“先总结上下文，再生成答案，最后做格式清洗”这类链式流程后，我意识到真正的重点已经不是单个 prompt，而是 workflow。用户面对的其实不是一次调用，而是一组有顺序的能力。

因此我在前端里加入了流程节点视图，让每一步的输入、输出和耗时都可见。哪怕用户不关心底层细节，这种透明度也会显著提升信任感。
