把任务拆给多个 Agent 之后，前端界面应该如何呈现进度与边界

多 Agent 并行会显著提升复杂任务的执行效率，但也会把原本简单的“过程反馈”问题变得更难。界面如果没有设计好，用户很容易陷入信息过载。

Multi-Agent Progress Frontend

第一层：任务树，而不是聊天流

当系统只有一个 Agent 时，对话流还能勉强承担任务进展展示的职责；但一旦进入多 Agent 场景，单一消息流会很快失控。不同子任务的启动、等待、完成与失败会交织在一起，用户几乎无法建立清晰心智模型。

所以我更偏向于用任务树作为一层主视图，把总任务、子任务、依赖关系和当前状态放到一个可折叠的结构里。

第二层：把日志留给需要的人

并不是所有用户都想看到工具调用细节，所以日志不应该直接占据主界面。我通常会把它放在二级面板或抽屉里，让重度用户按需展开。

主界面只保留对大多数用户最关键的信息：任务是否卡住、是否需要确认、是否已经产出可查看结果。

第三层：失败要能局部重试

如果一个子 Agent 执行失败，最糟糕的体验就是整个流程全部重来。好的界面应该告诉用户具体是哪一个节点失败、失败原因是什么，以及是否支持局部重试或替代方案。

  • 失败节点高亮，而不是整页报错。
  • 允许只重试当前步骤。
  • 必要时提供“改为人工继续”的接管入口。

多 Agent 的前端设计，本质上是在解决复杂系统的可解释性问题。只要信息分层做得足够清楚，复杂并不会天然导致混乱。
