Agent UI 的关键不是炫技，而是让用户理解“下一步会发生什么”

真正决定 Agent 产品体验上限的，往往不是模型能力本身，而是用户能否理解系统当前状态、预期动作与失败边界。

Agent UX Feedback Trust

黑盒感是 Agent 产品最大的阻力

普通对话产品的等待还能被理解成“模型在生成中”，但 Agent 不一样。它可能在搜索文件、拆任务、调用工具、等待返回。如果界面只给一个旋转中的 loading，用户会迅速失去方向感。

所以我越来越倾向于把 Agent 的执行过程显式拆开，让界面回答三个问题：现在在做什么、为什么这么做、接下来可能做什么。

状态反馈应该从单点变成连续流

我不再把状态设计成“待执行 / 执行中 / 已完成”这样过于粗糙的三段式，而是把 Agent 流程拆成一条连续的状态流。比如：分析需求、定位文件、修改代码、运行校验、整理结果。

当用户能看见这条流程线时，即使任务还没结束，也会更愿意等待，因为系统正在表达可理解的进展，而不是单纯地占用时间。

要提前暴露边界，而不是事后道歉

很多 Agent 产品只在失败后告诉用户“执行失败”。但真正好的体验应该在开始前就说清边界，例如：会修改哪些文件、是否会运行命令、是否需要联网、哪些结果需要人工确认。

  • 在任务启动前给出行动范围摘要。
  • 在关键步骤前展示即将执行的动作。
  • 在失败时给出可恢复的下一步，而不是只有报错。

这种设计看似增加了界面复杂度，但换来的是更强的安全感和更高的可控性。
